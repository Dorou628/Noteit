# Noteit MVP PRD

## 1. 文档信息
- 项目名称：Noteit
- 阶段：MVP
- 更新时间：2026-04-18
- 当前目标：先交付可联调、可测试、可持续迭代的内容社区后端

## 2. 产品目标
Noteit 是一个仿小红书的图文内容社区。MVP 目标不是一次性做完最终高并发架构，而是先跑通以下闭环：

- 内容发布与更新
- 首页 Feed 浏览
- 点赞、收藏、关注
- 个人主页与沉淀列表
- 图片/头像直传 OSS
- AI 摘要状态流转

## 3. MVP 范围与当前实现
### 3.1 MVP 业务范围
- 文章正文：当前存数据库
- 图片、头像：客户端直传 OSS
- 登录：当前支持明文密码登录，业务接口仍使用请求头模拟登录态
- 数据库：MySQL 单库
- 架构形态：模块化单体
- 开发方式：TDD，按模块闭环逐步推进

### 3.2 MVP 接口契约能力
- 发布文章
- 更新文章
- 获取文章详情
- 上传图片和头像到 OSS
- 查看首页 Feed
- 点赞 / 取消点赞
- 收藏 / 取消收藏
- 关注 / 取关
- 查看个人主页
- 查看“我赞过的”“我收藏的”

说明：
- 上述能力是 MVP 对外接口契约范围，具体返回格式以 `docs/API-MVP.md` 为准。
- 当前真实实现进度按 `docs/DELIVERY-PLAN-MVP.md` 推进；已完成能力、阶段占位能力和后续预留能力必须在 API 文档中显式标注。

### 3.3 当前已落地能力
- 请求头模拟登录边界
- MVP 明文密码登录接口
- 文章发布、更新、详情
- 上传任务创建、上传完成确认
- 首页 Feed 基础分页和作者筛选
- 点赞 / 取消点赞
- 收藏 / 取消收藏
- 文章详情返回真实点赞/收藏计数与当前用户互动状态
- 用户已发布文章列表
- “我赞过的”“我收藏的”沉淀列表
- 关注 / 取关
- 关注列表 / 粉丝列表
- 用户主页和文章作者视图返回真实 `followed` 状态
- 编辑个人信息（昵称、头像、简介）

### 3.4 当前阶段占位能力
- 暂无。新增占位能力时必须先同步 `docs/API-MVP.md`。

### 3.5 当前明确不做
- 不实现 JWT 双令牌登录
- 不接入 Redis / Kafka / Canal / Bitmap / ZSet
- 不做复杂推荐算法
- 不做完整消息中心、评论、搜索

## 4. 最终目标架构
以下是已经确定的最终技术方向，但本轮不集成，只保留演进空间：

- Feed：收件箱 + 发件箱，推拉结合
- 登录：JWT 双令牌
- Feed 查询：三级缓存架构
- 关注一致性：Outbox + Canal + Kafka
- 点赞状态：分片 Bitmap
- 收藏状态：ZSet
- 深分页：优先用游标化与读模型优化

## 5. 演进原则
- 当前实现优先可运行，不优先高并发
- 所有最终方案都要先在文档和抽象接口里留口
- 新代码不要继续加深对主表高频计数冗余的依赖
- 外部 API 在 MVP 阶段尽量稳定，内部实现后续替换

## 6. 关键产品规则
### 6.1 文章
- 标题必填
- 正文必填
- 当前详情接口直接返回正文 content，便于联调
- Feed 列表只返回 contentPreview，不返回正文全文
- 文章更新后摘要状态重置为 `PENDING`

### 6.2 点赞 / 收藏
- 必须幂等
- 当前 MVP 可以先用 MySQL 关系表实现
- 高频计数不冗余在实体主表是最终原则
- 当前表中的计数字段视为兼容字段，不再作为后续架构演进的核心依赖

### 6.3 关注
- 不能关注自己
- 关注 / 取关必须幂等
- 当前 MVP 先走数据库强一致主链路

### 6.4 登录
- 当前登录接口通过明文密码校验账号，登录成功后返回用户信息：
  - `userId`
  - `nickname`
  - `avatarUrl`
  - `authMode=HEADER_MOCK`
- 当前业务接口仍通过请求头模拟用户身份：
  - `X-User-Id`
  - `X-User-Nickname`
- 后续再切换到 JWT 双令牌

## 7. MVP 成功标准
- 前后端可以基于当前接口完成联调
- 核心接口均有自动化测试保护
- 文档、代码、库表状态一致
- 后续切换到最终架构时不需要推翻模块边界
